# Отбор точек при записи тега (`TagsApp.data_set`)

Перед отправкой в хранилище сервис **TagsApp** фильтрует входящие точки
`(время, значение, качество)`. Логика зависит от LDAP-атрибута **`prsMaxLineDev`**
и **`prsValueTypeCode`** (тип значения тега). Реализация:
`src/common/tag_max_line_dev.py`.

В Redis на ключе `{tagId}.{svc_name}` хранятся последние **принятые** значение и
качество: `prsLastAcceptedY`, `prsLastAcceptedQ` (для согласованности фильтра в
серии точек и между запросами).

## Общие правила (все типы)

1. **Смена качества** — если у новой точки поле качества `q` отличается от
   последнего принятого `prsLastAcceptedQ`, точка **всегда** передаётся в
   хранилище (независимо от значения и порога).
2. **Два значения `None`** — если и предыдущее принятое значение, и новое —
   `None`, и качество **не** изменилось по сравнению с последним принятым, точку
   **не** передаём (дубли «пустого» значения отсекаются). Если качество при этом
   сменилось, срабатывает правило 1.

## Типы `prsValueTypeCode`

| Код | Тип        | Поведение при `prsMaxLineDev` |
|-----|------------|-------------------------------|
| 0, 1 | int, double | Как раньше: при `> 0` отбрасываются точки с \|new − prev\| < порога; при `0` порог не действует, пишутся все точки (кроме общих правил выше). |
| 2, 4 | строка, JSON | При **`prsMaxLineDev == 0`** — каждая новая точка уходит в хранилище (с учётом общих правил). При **`prsMaxLineDev != 0`** — пишем только если **значение не равно** предыдущему (сравнение по смыслу для строки; для JSON — равенство после нормализации, в т.ч. разбор `bytes` как UTF-8 JSON при необходимости). |
| 5   | табличный | **`prsMaxLineDev` не используется** — на отбор точек влияют только общие правила (качество и пара `None`/`None`). |

Точки без третьего элемента (без `q`) считаются с **`q = None`**.

## Коннекторы (WebSocket / MQTT)

Данные от коннекторов публикуются с ключом маршрутизации
`prsTag.app_api.data_set.*` и обрабатываются **тем же** методом `TagsApp.data_set`,
что и запись через HTTP API (`TagsAppAPI` внутри пересылает на
`prsTag.app_api.data_set.*`). Отдельного обхода хранилища нет: сначала отбор
точек, затем `prsTag.app.data_set.<tagId>` в сервисы хранилищ.

Если в сообщении для одного тега приходит **массив точек** `data: [[x1,y1,q1], [x2,y2,q2], ...]`,
они просеиваются **по порядку**: состояние «последняя принятая пара (y, q)»
обновляется после каждой неотсеянной точки внутри этого массива (и стыкуется с
Redis между запросами). Поле `data` приводится к списку в
`coerce_tag_data_items_for_data_set` (`src/common/tag_data_points.py`), в том
числе если коннектор передал одну точку без обёртки в список.

Связанный код коннекторов: `src/services/connectors/app/connectors_app_svc.py`
(WebSocket), `src/services/connectors/app/connectors_mqtt_app_svc.py` (запись
качества в теги и т.п.).

## Связанные файлы

- `src/services/tags/app/tags_app_svc.py` — вызов фильтра и обновление Redis.
- `src/common/tag_data_points.py` — нормализация точки и приведение `data` к списку.
- Тесты: `tests/unit/test_tag_max_line_dev.py`, `tests/unit/test_tags_app_data_set_max_line_dev.py`.

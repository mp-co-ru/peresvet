Тревоги
~~~~~~~

Тревога в МПК-Пересвет — это узел иерархии класса ``prsAlert``, привязанный **к тегу**
(родитель в LDAP всегда ``prsTag``). Условия срабатывания, квитирование и снятие
тревоги обрабатывает сервис ``alerts_app``; создание и изменение узлов в модели —
через ``alerts_api_crud`` и конфигуратор. Состояние активных тревог (время
возникновения, квитирования) хранится в **Redis**; сообщения о значениях тегов
приходят по **AMQP** из обменника платформы (topic, имя задаётся в ``broker.name``
конфигурации, по умолчанию в примерах часто используется ``peresvet``).

Модель и сервисы
^^^^^^^^^^^^^^^^

* **Иерархия:** узел ``prsAlert`` создаётся **внутри** выбранного тега. Тот же тег
  является источником текущих значений для тревог типов «порог» и «задержка по
  диапазону» (см. ниже).
* **``alerts_app``** подписывается на ``prsTag.app.data_set.<id_тега>`` для тегов,
  у которых есть активные тревоги простых типов, и дополнительно — на инициаторы
  сложных тревог (теги и расписания). Публикует события ``alarm_on``,
  ``alarm_acked``, ``alarm_off``.
* **``alerts_app_api``** — HTTP-доступ к списку тревог и квитированию: префикс
  маршрутов ``/v1/alarms`` (GET — выборка по ``parentId`` и флагам; PUT — тело с
  ``id`` тревоги и ``x`` — время квитирования в микросекундах или ISO8601, см.
  реализацию сервиса).
* **Несколько экземпляров ``alerts_app``:** очередь потребления с фиксированным
  именем даёт *конкурирующее* потребление сообщений; состояние тревог общее в
  Redis, поэтому параллельные экземпляры не дублируют одну и ту же обработку одного
  сообщения для одной очереди.

Атрибут ``prsJsonConfigString``
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Конфигурация тревоги задаётся JSON-объектом в атрибуте ``prsJsonConfigString`` узла
``prsAlert``. Обязательно для всех видов: поле ``autoAck`` (логическое) — при
значении ``true`` после ``alarm_on`` сразу публикуется ``alarm_acked`` с тем же
временем ``x``.

Поле ``kind`` определяет тип логики (если не указано, используется ``simple``):

.. list-table::
   :header-rows: 1
   :widths: 18 82

   * - ``kind``
     - Назначение
   * - ``simple``
     - Порог по значению **тега-родителя** тревоги: сравнение с одним числом
       ``value`` и направлением ``high``.
   * - ``duration``
     - Тревога возникает, если значение тега-родителя **непрерывно** находится в
       замкнутом диапазоне ``[rangeLow, rangeHigh]`` не меньше времени
       ``durationMicroseconds`` (в **микросекундах**, в том же масштабе, что и
       метки времени в потоке данных тегов).
   * - ``complex``
     - Условие вычисляет **дочерний** узел ``prsMethod`` этой тревоги (тот же
       механизм инициаторов, что у методов на тегах: теги и расписания). В
       конфигурации указывается ``methodId`` — UUID этого метода. RPC-вызов
       метода должен возвращать признак «тревога активна» (истина: ``true``, число
       ≠ 0, строки ``"true"``, ``"1"`` и т.п.). Сервис ``methods_app`` методы с
       родителем ``prsAlert`` **не** обрабатывает — только ``alerts_app``.

**Простая тревога (``simple``)** — дополнительные поля:

* ``high`` — ``true``: тревога при ``значение >= value``; ``false``: при
  ``значение < value``.
* ``value`` — числовой порог (сравнение с приведением к ``float``).

**Тревога с задержкой по диапазону (``duration``)** — дополнительные поля:

* ``rangeLow``, ``rangeHigh`` — границы диапазона (включительно).
* ``durationMicroseconds`` — сколько **подряд** значение должно оставаться в
  диапазоне, чтобы сработала тревога. Пример: 3 минуты = ``180000000``.

Пока тревога уже активна (после ``alarm_on``), «условие активности» для снятия
считается выполненным, пока значение остаётся в диапазоне; при выходе за
пределы диапазона публикуется ``alarm_off``, накопление времени сбрасывается.

**Сложная тревога (``complex``)** — дополнительные поля:

* ``methodId`` — UUID дочернего ``prsMethod`` (родитель в иерархии — данная
  тревога). У метода настраиваются инициаторы (теги, расписания) в конфигураторе
  так же, как для вычислительного метода на теге.

События AMQP
^^^^^^^^^^^^

При срабатывании логики ``alerts_app`` публикует в topic-обменник сообщения с
телом JSON и ключами маршрутизации вида:

.. list-table::
   :header-rows: 1
   :widths: 28 72

   * - Событие
     - Routing key
   * - Возникновение тревоги
     - ``prsAlert.app.alarm_on.<entryUUID_тревоги>``
   * - Квитирование (вручную через API или автоматически при ``autoAck``)
     - ``prsAlert.app.alarm_acked.<entryUUID_тревоги>``
   * - Пропадание (условие снято)
     - ``prsAlert.app.alarm_off.<entryUUID_тревоги>``

**Тело сообщения** (во всех трёх случаях одинаковая структура):

.. code-block:: json

   {
     "alertId": "<uuid тревоги>",
     "x": 1234567890123456
   }

Здесь ``x`` — метка времени в **микросекундах** (как в точках данных тегов).

Подписка потребителя на обменнике платформы:

* на все тревоги: ``prsAlert.app.alarm_on.*``, ``prsAlert.app.alarm_acked.*``,
  ``prsAlert.app.alarm_off.*``;
* на одну тревогу: заменить ``*`` на конкретный UUID.

Конфигуратор Grafana
^^^^^^^^^^^^^^^^^^^^

Работа с моделью через встроенный конфигуратор описана в разделе
:doc:`configurator/configurator`. Ниже — только шаги, относящиеся к тревогам.

Создание тревоги
""""""""""""""""

#. В дереве модели выберите **тег**, к которому относится параметр контроля.
#. Нажмите кнопку создания тревоги (колокольчик / сущность тревога — в зависимости
   от версии панели).
#. Задайте имя (``cn``), описание, флаг активности ``prsActive``.
#. В блоке **«Конфигурация»** тревоги выберите **тип**:

   * **Порог** — простая тревога: сравнение ``>=`` или ``<``, порог ``value``,
     автоквитирование.
   * **Задержка (диапазон)** — нижняя и верхняя граница диапазона и поле
     длительности в **микросекундах** (например, ``180000000`` на три минуты),
     автоквитирование.
   * **Метод** — введите UUID дочернего метода (удобнее сначала создать под
     тревогой узел ``prsMethod``, настроить инициаторы и адрес метода, скопировать
     его ``entryUUID`` в поле конфигурации тревоги).

#. Нажмите **Записать**.

Для сложной тревоги сначала создайте дочерний **метод** у той же тревоги,
привяжите к нему инициаторы (теги/расписания) и параметры — по аналогии с методом
на теге. В JSON тревоги укажите ``methodId`` этого метода.
